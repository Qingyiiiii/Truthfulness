# 优化方案参考

> 本文件是 v0.1 Seed 的公开版后续方向。它只描述可复用的工程判断，不包含真实视频、标注内容、本地环境、账号材料或内部执行记录。

## 1. v0.1 已经证明什么

v0.1 Seed 已跑通“结构化主张 -> 证据字段 -> 审核状态 -> gold-only 训练面 -> 固定切分 -> baseline 评测”的最小闭环。现阶段的价值是验证数据契约、校验规则和评测管线，而不是证明真实性分类模型已经可用。

最需要正视的信号是：多数类会抬高 accuracy，少数标签的 F1 很低。因此下一阶段优先修复数据与评测，而不是直接堆叠更大模型。

## 2. 数据优先的下一步

1. 固定公开可用的数据授权与脱敏规则，原始视频、截图和账号上下文始终不进入 Git。
2. 将机器初筛和人工 gold 严格分层；只有双向可追溯、人工确认的 `gold_*` 记录能进入训练和评测。
3. 为每次数据发布保存 `dataset_version`、`schema_version`、`prompt_version`、来源批次和 split 版本。
4. 优先补齐少数类：误导、缺少上下文、证据不足、不可核查、产品体验/广告等；同时保留 hard negative。
5. 以主张为最小单元，避免同一视频相邻句或同一父主张泄漏到不同数据划分。

## 3. 系统演进方向

| 模块 | 当前 v0.1 | 后续建议 |
| --- | --- | --- |
| 内容理解 | 离线文本与受控媒体入口 | 增加 ASR、OCR、术语归一化和可回退的输入质量标记 |
| 主张抽取 | 原子主张 schema 与规则边界 | 建立可审核的 claim 生成与拆分规则，记录父子关系 |
| 证据检索 | Provider 接口、证据元数据与质量字段 | 接入可替换检索器，优先一手来源并保存证据快照元数据 |
| 判断 | 保守 verdict 与人工复核入口 | 把“证据不足”和“需要上下文”作为正常输出，避免二元真伪化 |
| 训练 | gold-only smoke baseline | 在冻结评测集和类别均衡后，再比较轻量分类器、检索排序与结构化 LLM 辅助 |

## 4. 评测门槛

在满足以下条件前，不应宣传模型可用：

- 有独立、冻结且来源隔离的测试集；
- 每个核心标签具备足够样本，报告 macro 指标、混淆矩阵和置信区间；
- 与多数类、规则基线和人工复核流程进行对比；
- 完成错误分析，区分转写错误、主张切分错误、检索错误、证据质量错误和判断错误；
- 复现同一数据/配置/切分下的结果，并保留版本记录。

## 5. 安全与发布原则

- 不绕过平台访问控制；只处理公开或明确授权的输入。
- Cookie、Token、媒体、截图、运行日志、完整标注和来源特定修复规则仅保留在本地私有工作区。
- LLM 只能辅助抽取、改写、排序或解释，不能替代外部证据。
- 高风险领域不输出可直接执行的医疗、法律或金融结论。

## 6. 建议里程碑

| 里程碑 | 完成条件 |
| --- | --- |
| M1：数据治理 | 可发布样例、版本字段、审核准入和泄漏检查固定 |
| M2：检索闭环 | 主张到证据的可追溯链路、来源分级和失败回退稳定 |
| M3：评测基线 | 冻结测试集、类别分析和可重复 benchmark 完成 |
| M4：模型比较 | 在 M3 通过后，比较分类、排序和结构化推理方案 |

v0.1 的正确定位是 M1/M2 的种子验证，而不是 M4 的模型成果。
